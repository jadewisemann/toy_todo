# API 추가 요청: 로그인 사용자 정보 조회 엔드포인트

## 1. 개요
프론트엔드에서 JWT 기반의 로그인 상태를 안정적으로 유지하고, 새로고침 시 사용자 세션을 복구하기 위해 현재 로그인된 사용자의 정보를 조회할 수 있는 API 추가를 요청드립니다.

## 2. 요청 사유 (Why we need this)
현재 프론트엔드는 로그인 성공 시 발급받은 JWT 토큰을 브라우저의 로컬 스토리지에 저장하여 인증 상태를 관리하고 있습니다. 하지만 사용자가 브라우저를 **새로고침(F5)** 하거나 새 탭에서 접속할 경우, 프론트엔드 메모리(React State)에 저장되어 있던 유저 상세 정보(`id`, `name` 등)가 초기화됩니다.

이 정보를 복구하기 위해 프론트엔드에서 토큰을 직접 디코딩하는 방식도 있지만, 아래와 같은 이유로 백엔드에서 정보를 조회해 오는 패턴이 아키텍처 및 보안 관점에서 훨씬 바람직합니다.

1. **실시간 유효성 및 보안 검증**
   - 서버에서 계정 정지 처리를 하거나 유저가 비밀번호를 변경하여 기존 토큰을 만료시켜야 하는 상황을 가정해 보겠습니다. 
   - 프론트엔드에서 토큰만 디코딩하면 만료 시간을 넘기지 않는 이상 해당 유저를 계속 로그인된 것으로 간주하게 됩니다.
   - API를 통해 서버에 접근하면 서버가 토큰의 유효성을 확실히 검증하여 즉시 로그아웃(401 에러) 처리를 할 수 있습니다.
2. **데이터 최신화 보장**
   - 닉네임이나 프로필 등 유저 정보가 변경되었을 때, 낡은 토큰 데이터를 읽지 않고 항상 DB 기반의 최신 상태를 화면에 렌더링할 수 있습니다.
3. **JWT Payload 경량화**
   - 프론트엔드 디코딩에 의존하면 토큰 Payload 내에 너무 많은 유저 정보를 담아야 하여 네트워크 대역폭 낭비가 발생합니다. 토큰에는 최소 식별자만 담고, 정보는 API로 분리하는 것이 좋습니다.

## 3. API 명세 제안 (Specification)

### `GET /api/accounts/me/`

#### Request
- **Method**: `GET`
- **Headers**:
  - `Authorization`: `Bearer <jwt_token>`

#### Response - 성공 시 (200 OK)
```json
{
  "user": {
    "id": "user123",
    "name": "홍길동"
  }
}
```

#### Response - 실패 시 (401 Unauthorized)
토큰이 없거나, 유효하지 않은 토큰이거나, 만료되었거나, 차단/탈퇴한 유저인 경우 401 에러를 반환해 주시면 됩니다.
```json
{
  "message": "인증에 실패했습니다. 유효하지 않은 토큰입니다." 
}
```

## 4. 프론트엔드 연동 계획
해당 API가 제공되면 프론트엔드에서는 앱의 최상위(Root) 레벨에서 브라우저 로드 시 이 API를 가장 먼저 호출합니다.
- `200 OK` 응답 시: 반환된 유저 정보를 상태 관리에 저장하고 정상적인 서비스를 제공합니다.
- `401 Unauthorized` 응답 시: 만료된 세션으로 판단하여 로컬 스토리지의 토큰을 즉시 폐기하고 로그인 페이지로 리다이렉트합니다.
